## Key Decisions Made

- **All-JS-click approach in Playwright:** `page.evaluate(() => el.click())` bypasses Playwright's pointer-event interception checks. The fixed side panels (`z-40`) and AI Chat overlay (`z-30`) block Playwright's normal `.click()` method.
- **`ready_nodes` proxy logic:** List API intentionally returns `ready_nodes: 0` (comment in `aks_endpoints.py`: "Not available in list view; accurate count available in detail endpoint"). The UI fix uses `node_pools_ready === node_pool_count` as a reliable proxy — if all node pools report ready, assume all nodes are ready. This avoids a false red/yellow badge showing `0/3`.
- **Panel close on tab switch:** Root cause of the stacking bug was that `handleTabChange` in `DashboardPanel.tsx` only called `setActiveTab` — it never closed other panels. Both `vmssDetailOpen` and `aksDetailOpen` could be `true` at the same time, causing both panels to render at the same `fixed top-0 right-0 z-40` position. The VMSS panel (rendered first in JSX) occluded the AKS panel entirely.

## Context / Mental Model

The app has two levels of tab navigation:
1. **Main tabs** (`[role="tab"]`): VMs, VMSS, AKS, Networking, Cost, Alerts — these switch the main content panel
2. **Detail panel buttons** (`<button>` elements inside `div.fixed.top-0.right-0.h-full.z-40`): Overview, Instances/Node Pools/Workloads, Metrics, Scaling, AI Chat — appear when you click a row

The `DashboardPanel.tsx` fix adds 6 state resets to `handleTabChange` so all detail panels are closed whenever the user switches main tabs. This is the correct UX behavior — the detail panel is contextual to the tab you're on.

The `AKSTab.tsx` fix is a UI-only workaround for an intentional API limitation. The backend list endpoint comment says `ready_nodes` isn't populated for performance reasons. The fix makes the badge green (showing `3/3`) when all node pools are ready, rather than misleadingly red (showing `0/3`).
