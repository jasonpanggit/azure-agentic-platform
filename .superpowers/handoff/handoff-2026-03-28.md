## Key Decisions Made

- **`use_placeholder_image = true` (default):** Container Apps require an image to provision. Using `mcr.microsoft.com/azuredocs/containerapps-helloworld:latest` as placeholder until CI/CD builds real images. Flip to `false` in prod `main.tf` module call once images are pushed.
- **`provision_fabric_dataplane = false` (default):** Fabric workspace/items are data-plane resources accessible only via Fabric REST API with user auth â€” not ARM. SP cannot provision them. Must be done manually or via a separate pipeline with user credentials.
- **`Cosmos DB Operator` in ARM RBAC:** `Cosmos DB Built-in Data Contributor` is a Cosmos-API role, not ARM. ARM role assignment uses `Cosmos DB Operator` (management-plane). Data-plane access granted separately via `scripts/grant-cosmos-rbac.sh` using `az cosmosdb sql role assignment create`.
- **SP has `Contributor` + `User Access Administrator`:** Needed UAA to allow Terraform to create role assignments. Granted manually as subscription Owner.
- **Event Hub `network_rulesets` without `public_network_access_enabled = false`:** Azure API rejects combining both; `default_action = "Deny"` already enforces private-only access.
- **Fabric capacity name `fcaapprod`:** Azure Fabric schema enforces `^[a-z][a-z0-9]*$` (no hyphens). Renamed from `fc-aap-prod`.

## Watch Out For

- **Cosmos RBAC script** (`scripts/grant-cosmos-rbac.sh`) uses role ID `00000000-0000-0000-0000-000000000002` â€” this is the well-known Cosmos Built-in Data Contributor ID. Safe to re-run idempotently.
- **`credentials.tfvars` contains real secrets** â€” SP client secret and postgres password. Already in `.gitignore`. Never commit.
- **`terraform/envs/prod/credentials.tfvars`** is the working copy (not repo root). Always `cd terraform/envs/prod` before running terraform commands.
- **State lock** â€” if a previous apply was killed, run `terraform force-unlock -force <LOCK_ID>` before retrying.
- **RBAC propagation** â€” Azure role assignments can take 1-5 minutes to propagate. If Cosmos connections fail immediately after grant, wait and retry.
- **Fabric dataplane resources** â€” `module.fabric.azapi_resource.fabric_workspace` etc. have `count = 0` until `provision_fabric_dataplane = true`. Don't enable this via SP â€” it will fail with 404 `InvalidResourceType`.
- **`azurerm_eventhub_consumer_group` and `azurerm_eventhub_namespace_authorization_rule`** still use deprecated `namespace_name` (azurerm 4.x doesn't support `namespace_id` for these yet). Will need updating when upgrading to azurerm v5.
