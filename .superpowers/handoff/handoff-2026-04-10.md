## Key Decisions Made

- **Separate batch endpoint (not inline enrichment):** Keep VM list fast — don't add EOL lookups to the main `/api/v1/vms` call. Fetch EOL separately after VM list loads.
- **PostgreSQL cache first, endoflife.date API fallback:** Use existing `eol_cache` table, fall back to direct API call if not cached.
- **Column header "EOL Date"** — show formatted date or "EOL" badge if already past EOL, or "—" if unknown.

---

## Watch Out For

- **api-gateway has its own DB connection** — it does NOT share the EOL agent's `get_cached_eol()` function. The api-gateway will need its own direct `asyncpg` query to PostgreSQL. Look at how `main.py` sets up the DB pool (it already does for other tables).
- **OS name parsing:** ARG returns `os_name` like `"Windows Server 2025 Standard"` or `"Windows Server 2019 Datacenter"`. Need to strip the suffix ("Standard", "Datacenter") to get the product key. The product map in `agents/eol/tools.py` uses lowercase keys like `"windows server 2025"`.
- **Cache may be empty initially** — `eol_cache` is only populated when the EOL agent runs scans. If the cache is empty, fall back to calling `endoflife.date` API directly from the endpoint.
- **endoflife.date API format:** `GET https://endoflife.date/api/{product}/{cycle}.json` → `{ "eol": "2026-10-14", "lts": false, ... }`. For Windows Server, use `ms-lifecycle` (Microsoft Product Lifecycle API), not endoflife.date.
- **ACR builds:** always use `--agent-pool aap-builder-prod`. Force new Container Apps revision with `--set-env-vars "DEPLOY_TIME=<label>"` if image tag doesn't change.

## Context / Mental Model

### EOL infrastructure (already exists, don't rewrite)

**`agents/eol/tools.py`** has:
- `PRODUCT_MAP` dict: lowercase OS name → `(source, slug)` e.g. `"windows server 2025" → ("ms-lifecycle", "windows-server-2025")`
- `MS_PRODUCTS` set: products that use the MS Lifecycle API
- `normalize_product_slug(product_name, version)` → `(source, slug, cycle)` — converts OS name to API-ready form
- `get_cached_eol(product, version)` / `set_cached_eol(...)` — async PostgreSQL cache functions (use the agents' own DB connection, NOT usable from api-gateway directly)

**PostgreSQL `eol_cache` table** (in the shared Postgres instance api-gateway also uses):
```sql
eol_cache (
  product TEXT,       -- e.g. "windows-server-2025"
  version TEXT,       -- e.g. "2025"
  eol_date DATE,
  is_eol BOOLEAN,
  lts BOOLEAN,
  latest_version TEXT,
  source TEXT,        -- "ms-lifecycle" or "endoflife.date"
  expires_at TIMESTAMPTZ
)
-- UNIQUE on (product, version, source)
-- INDEX on (product, version, expires_at)
```

### OS name normalization logic (implement in api-gateway, don't import from agents)

```python
def _parse_os_for_eol(os_name: str) -> tuple[str, str] | None:
    """Returns (product_slug, cycle) or None if unrecognized."""
    lower = os_name.lower()
    # Windows Server: strip "standard", "datacenter", "essentials" suffix
    for year in ["2025", "2022", "2019", "2016", "2012 r2", "2012", "2008 r2", "2008"]:
        if f"windows server {year}" in lower:
            return f"windows-server-{year.replace(' ', '-')}", year.replace(' ', '-')
    # Ubuntu: "Ubuntu 22.04 LTS" → ("ubuntu", "22.04")
    # Add more as needed
    return None
```
