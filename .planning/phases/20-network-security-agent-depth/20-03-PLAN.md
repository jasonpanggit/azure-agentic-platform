## Threat Model (ASVS L1)

### Trust Boundaries
- SRE agent runs inside a Container App with system-assigned managed identity
- Reader + Monitoring Reader RBAC scoped across all subscriptions (enforced by Terraform)
- All tools are strictly read-only — no write operations
- `propose_remediation` generates proposals only; REMEDI-001 enforced (no execution without human approval)

### Identified Threats
1. **Cross-domain correlation info aggregation:** `correlate_cross_domain` aggregates data from multiple sources
   - Mitigation: Tool only combines data already accessible to the SRE agent's Reader role. No elevation of privilege. Data stays within the agent context.
2. **Change Analysis service availability:** `azure-mgmt-changeanalysis` may be unavailable in some regions
   - Mitigation: Graceful degradation — returns empty results with warning if service unavailable, never raises.
3. **Large metrics result sets from Monitor:** Querying availability across long timespans could return massive datasets
   - Mitigation: Default timespan is PT24H; metrics are aggregated (not raw points). Reasonable bounds.
4. **Advisor recommendations could reveal cost/performance details**
   - Mitigation: Data flows only to agent, not exposed externally. Reader role is read-only.

### Residual Risk
- **Low:** `azure-mgmt-resourcehealth` uses preview SDK (`1.0.0b6`). API may change. Acceptable — tool handles exceptions gracefully.
- **Low:** `azure-mgmt-changeanalysis` may have limited coverage. Tool returns empty results when unavailable.
</threat_model>

<action>
Add 4 new `@ai_function` tools to `agents/sre/tools.py`:

**Tool 4: `query_service_health(subscription_id, event_type=None)`**
- SDK: `ResourceHealthMgmtClient(credential, subscription_id).events.list_by_subscription_id()`
- Filter: if `event_type` is not None (one of "ServiceIssue", "PlannedMaintenance", "HealthAdvisory", "SecurityAdvisory"), filter results
- For each event: `event_type`, `summary`, `status` ("Active"/"Resolved"), `impact_start_time` (isoformat if not None), `last_update_time` (isoformat if not None), `header` (if available), `level`
- Cap at 50 events max
- Return: `{"subscription_id": ..., "event_type_filter": event_type, "events": [...], "event_count": int, "active_count": int (where status == "Active"), "query_status": "success"}`

**Tool 5: `query_advisor_recommendations(subscription_id, category=None)`**
- SDK: `AdvisorManagementClient(credential, subscription_id).recommendations.list()`
- Filter: if `category` is not None (one of "HighAvailability", "Security", "Performance", "Cost", "OperationalExcellence"), filter results where `rec.category == category`
- For each recommendation: `category`, `impact` ("High"/"Medium"/"Low"), `impacted_field`, `impacted_value`, `problem` (from `rec.short_description.problem` if available), `solution` (from `rec.short_description.solution` if available), `resource_id` (from `rec.resource_metadata.resource_id` if available)
- Cap at 100 recommendations max
- Return: `{"subscription_id": ..., "category_filter": category, "recommendations": [...], "recommendation_count": int, "high_impact_count": int, "query_status": "success"}`

**Tool 6: `query_change_analysis(subscription_id, timespan_hours=2, resource_group=None)`**
- SDK: `AzureChangeAnalysisManagementClient(credential, subscription_id)`
- Compute `start_time` and `end_time` from `timespan_hours`
- If `resource_group` is provided: `client.changes.list_changes_by_resource_group(resource_group, start_time, end_time)`
- Else: `client.changes.list_changes_by_subscription(start_time, end_time)`
- For each change (cap at 100): `resource_id`, `change_type` ("Add"/"Remove"/"Update"), `time_stamp` (isoformat), `initiated_by_list` (list of strings if available), `property_changes` (list of `{"property_name": ..., "old_value": ..., "new_value": ...}` — up to 10 per change)
- Return: `{"subscription_id": ..., "timespan_hours": ..., "resource_group_filter": resource_group, "changes": [...], "change_count": int, "query_status": "success"}`

**Tool 7: `correlate_cross_domain(subscription_id, resource_id, timespan_hours=2)`**
- This is a composite tool — it calls the other SRE tools internally to build a correlation view:
  1. Call `query_service_health(subscription_id)` to check for active platform events
  2. Call `query_change_analysis(subscription_id, timespan_hours)` to find recent changes
  3. Call `query_availability_metrics(resource_id, f"PT{timespan_hours}H")` to check resource availability
  4. Call `query_advisor_recommendations(subscription_id)` and filter to those affecting `resource_id`
- Each sub-call is wrapped in try/except — partial failures are captured, not fatal
- Build correlation dict:
  - `platform_events`: active service health events
  - `recent_changes`: changes in timespan, sorted by timestamp
  - `availability_impact`: availability data for the resource
  - `relevant_recommendations`: advisor recs for the resource
  - `correlation_summary`: text summary: "Found X platform events, Y changes in last Zh, availability at N%"
- Return: `{"subscription_id": ..., "resource_id": ..., "timespan_hours": ..., "platform_events": [...], "recent_changes": [...], "availability_impact": {...}, "relevant_recommendations": [...], "correlation_summary": str, "query_status": "success"}`

All tools follow the established pattern (instrument_tool_call, start_time, duration_ms, never raise, structured error dicts).
</action>
