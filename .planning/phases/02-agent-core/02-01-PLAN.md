# Plan 02-01: Agent Specs + CI Lint Gate + Terraform Identity/RBAC

> **Wave 1** — Must complete before any agent implementation code (Waves 2-5).

## must_haves

1. All 7 `docs/agents/{name}-agent.spec.md` files exist with Persona, Goals, Workflow steps, Tool permissions (explicit allowlist, no wildcards), Safety constraints, and Example flows sections.
2. CI lint gate (`agent-spec-lint.yml`) fails a PR that adds `agents/{name}/*.py` files without a corresponding `docs/agents/{name}-agent.spec.md`.
3. Terraform `agent-apps` module defines 7 Container Apps (one per agent) + 1 API gateway Container App, each with `identity { type = "SystemAssigned" }`, and outputs all 8 `principal_id` values.
4. Terraform `rbac` module creates `azurerm_role_assignment` resources mapping each agent's `principal_id` to its domain-specific Azure built-in role at the correct subscription/resource-group scope.
5. Each agent's `principal_id` output is preserved as `agent_entra_ids` map output from the `agent-apps` module for Phase 7 Agent 365 auto-discovery.
6. A new `sessions` Cosmos DB container is provisioned in the databases module with partition key `/incident_id` for budget tracking (AGENT-007 prep).

Create `docs/agents/orchestrator-agent.spec.md` with the following sections:

**Persona:** The Orchestrator is the central dispatcher for all Azure infrastructure incidents. It classifies incoming incidents by domain, routes to the correct specialist agent, and manages cross-domain escalations. It does NOT diagnose or remediate directly.

**Goals:**
1. Classify every incident by domain (compute/network/storage/security/arc/sre) using payload `domain` field or LLM classification when ambiguous
2. Route to the correct domain AgentTarget via HandoffOrchestrator handoff
3. Handle cross-domain re-routing when a domain agent returns `needs_cross_domain: true`
4. Ensure all messages use the typed JSON envelope (AGENT-002)

**Workflow steps:**
1. Receive incident from Foundry thread (created by API gateway)
2. Parse `IncidentMessage` envelope; validate `correlation_id`, `thread_id`, `source_agent`, `target_agent`, `message_type`
3. If `domain` field is present and unambiguous, route directly to domain agent
4. If `domain` is absent or ambiguous, classify using LLM analysis of `affected_resources`, `detection_rule`, and `kql_evidence`
5. Create handoff message with `message_type: "incident_handoff"` to target domain agent
6. Monitor domain agent response; if `needs_cross_domain: true`, re-route to secondary domain agent
7. Aggregate final diagnosis and return to thread

**Tool permissions (explicit allowlist):**
- NO Azure MCP Server tools (Orchestrator does not query resources directly)
- NO write operations of any kind
- Foundry thread management only (create_message, list_messages)

**Safety constraints:**
- MUST NOT query Azure resources directly; all resource queries delegated to domain agents
- MUST NOT propose or execute any remediation action
- MUST NOT skip classification step for any incident
- Reader RBAC only on platform subscription

**Example flows:**
- Flow 1: CPU spike alert -> domain=compute -> handoff to compute-agent -> diagnosis returned
- Flow 2: Ambiguous NSG alert -> LLM classifies as network -> handoff to network-agent -> agent discovers root cause is storage throttling -> returns needs_cross_domain:true -> re-route to storage-agent
- Flow 3: Arc server disconnection -> domain=arc -> handoff to arc-agent -> arc-agent returns "Arc capabilities pending Phase 3" stub response
</action>

<read_first>

<action>
Create `.github/workflows/agent-spec-lint.yml`:

```yaml
name: Agent Spec Lint

on:
  pull_request:
    paths:
      - 'agents/**/*.py'
      - 'docs/agents/**'

jobs:
  check-specs:
    name: Verify Agent Spec Files
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Verify agent specs exist
        run: |
          EXIT_CODE=0
          # Find all agent directories (exclude 'shared')
          for agent_dir in agents/*/; do
            agent_name=$(basename "$agent_dir")
            # Skip shared utilities directory
            if [ "$agent_name" = "shared" ]; then
              continue
            fi
            # Check if directory contains Python files
            if ls "$agent_dir"*.py 2>/dev/null 1>&2 || ls "$agent_dir"**/*.py 2>/dev/null 1>&2; then
              spec_file="docs/agents/${agent_name}-agent.spec.md"
              if [ ! -f "$spec_file" ]; then
                echo "::error file=${spec_file}::Missing spec file: ${spec_file} (required by AGENT-009)"
                EXIT_CODE=1
              else
                echo "OK: ${spec_file} exists"
                # Verify required sections exist
                for section in "## Persona" "## Goals" "## Workflow" "## Tool Permissions" "## Safety Constraints" "## Example Flows"; do
                  if ! grep -q "$section" "$spec_file"; then
                    echo "::error file=${spec_file}::Missing required section: ${section}"
                    EXIT_CODE=1
                  fi
                done
              fi
            fi
          done
          if [ $EXIT_CODE -eq 0 ]; then
            echo "All agent spec files present and valid."
          fi
          exit $EXIT_CODE
```

The workflow triggers on PRs modifying any file under `agents/` with `.py` extension or any file under `docs/agents/`. For each `agents/{name}/` directory containing `.py` files (excluding `agents/shared/`), it checks that `docs/agents/{name}-agent.spec.md` exists and contains all 6 required sections.
</action>

      }

      for sub_id in var.all_subscription_ids : [

          key                  = "sre-reader-${sub_id}"

        scope                = rg_id
      }

        principal_id         = principal_id
        role_definition_name = "Cosmos DB Built-in Data Contributor"
        scope                = "/subscriptions/${var.platform_subscription_id}"
      }
