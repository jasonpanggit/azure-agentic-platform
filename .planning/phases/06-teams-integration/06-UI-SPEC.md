# Phase 6 — Adaptive Card Design Contract

> Visual and interaction contract for Teams bot Adaptive Cards. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Note:** This phase has NO web UI components. All "UI" is Microsoft Teams Adaptive Cards (v1.5) rendered by the Teams platform. shadcn, Tailwind, and Fluent UI React do NOT apply. The design contract defines card schemas, semantic color usage, typography within the Adaptive Card system, card layout patterns, copywriting, and action contracts.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | Adaptive Cards v1.5 |
| Preset | not applicable |
| Component library | Microsoft Teams Adaptive Card renderer (native) |
| Icon library | none (Adaptive Cards use text-based semantic indicators, not custom icons) |
| Font | Segoe UI (inherited from Teams; not configurable in Adaptive Cards) |

**Adaptive Card constraints:**
- Maximum payload size: 28 KB per card
- Maximum actions per card: 6
- Schema: `http://adaptivecards.io/schemas/adaptive-card.json`
- Version: `1.5`
- Cards must render correctly in Teams Light, Dark, and High Contrast themes — use only semantic color tokens (never hardcoded hex)

---

## Spacing Scale

Adaptive Cards use a predefined spacing enum — not pixel values. All card elements MUST use these tokens:

| Token | Adaptive Card Value | Usage |
|-------|-------------------|-------|
| none | `"spacing": "none"` | Adjacent elements with no gap (e.g., heading + subheading) |
| small | `"spacing": "small"` | Tight grouping within a section (label above value) |
| default | `"spacing": "default"` | Standard element-to-element spacing |
| medium | `"spacing": "medium"` | Between logical sections within a card |
| large | `"spacing": "large"` | Major section breaks |
| extraLarge | `"spacing": "extraLarge"` | Top-level section separation (rarely used) |
| padding | `"spacing": "padding"` | Match card padding (outermost container only) |

**Card-level rules:**
- Heading to first body element: `"spacing": "small"`
- Between FactSet and actions: `"spacing": "medium"`
- Between body TextBlocks: `"spacing": "default"`
- Use `"separator": true` between major card sections (e.g., facts → actions)

Exceptions: none — all cards use the Adaptive Card spacing enum exclusively.

---

## Typography

Adaptive Cards use a predefined size enum. Font family and line-height are NOT configurable (Teams controls them).

| Role | Size Token | Weight Token | Usage |
|------|-----------|-------------|-------|
| Card Title | `"size": "medium"` | `"weight": "bolder"` | Primary heading per card (1 per card max) |
| Section Label | `"size": "default"` | `"weight": "bolder"` | FactSet titles, section headers |
| Body | `"size": "default"` | `"weight": "default"` | Description text, status messages |
| Metadata | `"size": "small"` | `"weight": "default"` | Timestamps, secondary info, expiry timers |

**Rules:**
- Never use `"size": "extraLarge"` — it is disproportionate in Teams chat cards
- Never use `"size": "large"` for card titles — `"medium"` is the correct card heading size in Teams
- All descriptive text MUST set `"wrap": true`
- FactSet `"title"` values are auto-bolded by the renderer; do not add `"weight": "bolder"` in FactSet items

---

## Color

Adaptive Cards use semantic color tokens that adapt to the user's Teams theme (Light, Dark, High Contrast). **Never use hardcoded hex values.**

| Role | Token | Usage |
|------|-------|-------|
| Default text | `"color": "default"` | All standard body text, descriptions, metadata |
| Accent | `"color": "accent"` | Informational highlights (subscription name, thread ID links) |
| Good (success) | `"color": "good"` | Success outcomes, "Approved" status, healthy resource state |
| Warning | `"color": "warning"` | Medium-risk labels, escalation reminder headers, "Pending" status |
| Attention (danger) | `"color": "attention"` | Critical/high-risk labels, "Rejected" status, failure outcomes, error states |

**Container styles:**

| Style | Usage |
|-------|-------|
| `"style": "default"` | Standard card background |
| `"style": "emphasis"` | Highlighted section (e.g., risk summary in approval cards) |
| `"style": "attention"` | Critical alert cards only (sev0/sev1) |
| `"style": "warning"` | Escalation reminder cards |
| `"style": "good"` | Success outcome cards |

**Action button styles:**

| Style | Usage |
|-------|-------|
| `"style": "positive"` | Approve buttons only |
| `"style": "destructive"` | Reject buttons only |
| (no style / default) | Investigate (Action.OpenUrl), informational actions |

**Accent reserved for:** subscription name in alert cards, thread ID references, Web UI deep links in text. Never for headings or body text.

## Card Type Contracts

Phase 6 defines exactly 4 card types, each triggered by the api-gateway calling `POST /teams/internal/notify` with a `card_type` discriminator.

### Card Type 1: Alert Card (`card_type: "alert"`)

**Trigger:** New incident created in Cosmos DB (api-gateway fires notify).
**Requirement:** TEAMS-002
**Source pattern:** Extends the existing `_build_adaptive_card()` structure from `services/api-gateway/teams_notifier.py`.

```json
{
  "$schema": "http://adaptivecards.io/schemas/adaptive-card.json",
  "type": "AdaptiveCard",
  "version": "1.5",
  "body": [
    {
      "type": "TextBlock",
      "text": "Alert: {alert_title}",
      "weight": "Bolder",
      "size": "Medium",
      "color": "{severity_color}"
    },
    {
      "type": "FactSet",
      "facts": [
        { "title": "Resource", "value": "{resource_name}" },
        { "title": "Severity", "value": "{severity}" },
        { "title": "Subscription", "value": "{subscription_name}" },
        { "title": "Time", "value": "{timestamp_iso}" },
        { "title": "Domain", "value": "{domain}" }
      ]
    }
  ],
  "actions": [
    {
      "type": "Action.OpenUrl",
      "title": "Investigate in Web UI",
      "url": "{WEB_UI_PUBLIC_URL}/incidents/{incident_id}"
    }
  ]
}
```

**Color mapping:**
| Severity | Title Color | Container Style |
|----------|------------|-----------------|
| Sev0 (Critical) | `"attention"` | `"attention"` |
| Sev1 (Error) | `"attention"` | `"default"` |
| Sev2 (Warning) | `"warning"` | `"default"` |
| Sev3 (Informational) | `"default"` | `"default"` |

**Copywriting:**
- Title format: `"Alert: {alert_title}"` (e.g., `"Alert: High CPU on vm-prod-01"`)
- FactSet values are direct data — no sentence formatting
- Timestamp format: ISO 8601 with timezone (e.g., `"2026-03-27T14:30:00Z"`)

### Card Type 2: Approval Card (`card_type: "approval"`)

**Trigger:** High-risk remediation proposal created (REMEDI-002 triggers HITL gate).
**Requirement:** TEAMS-003
**Source pattern:** Direct port of `_build_adaptive_card()` from `services/api-gateway/teams_notifier.py` — preserves the Action.Http + FactSet structure per D-08.

```json
{
  "$schema": "http://adaptivecards.io/schemas/adaptive-card.json",
  "type": "AdaptiveCard",
  "version": "1.5",
  "body": [
    {
      "type": "TextBlock",
      "text": "Remediation Approval Required ({risk_level})",
      "weight": "Bolder",
      "size": "Medium",
      "color": "{risk_color}"
    },
    {
      "type": "TextBlock",
      "text": "{proposal_description}",
      "wrap": true,
      "spacing": "small"
    },
    {
      "type": "FactSet",
      "spacing": "medium",
      "facts": [
        { "title": "Target", "value": "{target_resources_csv}" },
        { "title": "Impact", "value": "{estimated_impact}" },
        { "title": "Risk Level", "value": "{risk_level}" },
        { "title": "Reversibility", "value": "{reversibility}" },
        { "title": "Expires", "value": "{expires_at_iso}" }
      ]
    }
  ],
  "actions": [
    {
      "type": "Action.Http",
      "title": "Approve",
      "method": "POST",
      "url": "{API_GATEWAY_PUBLIC_URL}/api/v1/approvals/{approval_id}/approve",
      "body": "{\"decided_by\": \"{{userName}}\"}",
      "style": "positive"
    },
    {
      "type": "Action.Http",
      "title": "Reject",
      "method": "POST",
      "url": "{API_GATEWAY_PUBLIC_URL}/api/v1/approvals/{approval_id}/reject",
      "body": "{\"decided_by\": \"{{userName}}\"}",
      "style": "destructive"
    }
  ]
}
```

**Color mapping:**
| Risk Level | Title Color |
|------------|------------|
| critical | `"attention"` |
| high | `"warning"` |

**Copywriting:**
- Title format: `"Remediation Approval Required (CRITICAL)"` or `"Remediation Approval Required (HIGH)"`
- Description: verbatim `proposal.description` from the remediation payload — always `wrap: true`
- Target: comma-separated resource IDs (e.g., `"vm-prod-01, vm-prod-02"`)
- Expires: ISO 8601 timestamp (e.g., `"2026-03-27T15:00:00Z"`)
- Approve button label: `"Approve"` (never "Yes" or "OK")
- Reject button label: `"Reject"` (never "No" or "Deny")

### Card Type 3: Outcome Card (`card_type: "outcome"`)

**Trigger:** Approved remediation action completes execution (success or failure).
**Requirement:** TEAMS-006

```json
{
  "$schema": "http://adaptivecards.io/schemas/adaptive-card.json",
  "type": "AdaptiveCard",
  "version": "1.5",
  "body": [
    {
      "type": "TextBlock",
      "text": "Remediation {outcome_status}",
      "weight": "Bolder",
      "size": "Medium",
      "color": "{outcome_color}"
    },
    {
      "type": "TextBlock",
      "text": "{action_description}",
      "wrap": true,
      "spacing": "small"
    },
    {
      "type": "FactSet",
      "spacing": "medium",
      "facts": [
        { "title": "Status", "value": "{outcome_status}" },
        { "title": "Duration", "value": "{duration_seconds}s" },
        { "title": "Resource State", "value": "{resulting_resource_state}" },
        { "title": "Approved By", "value": "{approver_upn}" },
        { "title": "Executed At", "value": "{executed_at_iso}" }
      ]
    }
  ],
  "actions": [
    {
      "type": "Action.OpenUrl",
      "title": "View Details in Web UI",
      "url": "{WEB_UI_PUBLIC_URL}/incidents/{incident_id}"
    }
  ]
}
```

**Color mapping:**
| Outcome | Title Color | Container Style |
|---------|------------|-----------------|
| Succeeded | `"good"` | `"good"` |
| Failed | `"attention"` | `"attention"` |
| Aborted (stale_approval) | `"warning"` | `"warning"` |

**Copywriting:**
- Success title: `"Remediation Succeeded"`
- Failure title: `"Remediation Failed"`
- Abort title: `"Remediation Aborted"`
- Duration format: integer seconds with `"s"` suffix (e.g., `"12s"`)
- Resource state: verbatim from ARM resource health (e.g., `"Running"`, `"Stopped"`, `"Degraded"`)

### Card Type 4: Reminder Card (`card_type: "reminder"`)

**Trigger:** Escalation scheduler detects unacted approval older than `ESCALATION_INTERVAL_MINUTES` (default: 15 min).
**Requirement:** TEAMS-005
**Source:** D-17 in CONTEXT.md.

```json
{
  "$schema": "http://adaptivecards.io/schemas/adaptive-card.json",
  "type": "AdaptiveCard",
  "version": "1.5",
  "body": [
    {
      "type": "Container",
      "style": "warning",
      "items": [
        {
          "type": "TextBlock",
          "text": "\u26a0\ufe0f Reminder: Approval Required",
          "weight": "Bolder",
          "size": "Medium",
          "color": "warning"
        }
      ]
    },
    {
      "type": "TextBlock",
      "text": "{original_action_description}",
      "wrap": true,
      "spacing": "small"
    },
    {
      "type": "FactSet",
      "spacing": "medium",
      "facts": [
        { "title": "Target", "value": "{target_resources_csv}" },
        { "title": "Risk Level", "value": "{risk_level}" },
        { "title": "Pending Since", "value": "{created_at_iso}" },
        { "title": "Expires In", "value": "{remaining_minutes} minutes" }
      ]
    }
  ],
  "actions": [
    {
      "type": "Action.Http",
      "title": "Approve",
      "method": "POST",
      "url": "{API_GATEWAY_PUBLIC_URL}/api/v1/approvals/{approval_id}/approve",
      "body": "{\"decided_by\": \"{{userName}}\"}",
      "style": "positive"
    },
    {
      "type": "Action.Http",
      "title": "Reject",
      "method": "POST",
      "url": "{API_GATEWAY_PUBLIC_URL}/api/v1/approvals/{approval_id}/reject",
      "body": "{\"decided_by\": \"{{userName}}\"}",
      "style": "destructive"
    },
    {
      "type": "Action.OpenUrl",
      "title": "View in Web UI",
      "url": "{WEB_UI_PUBLIC_URL}/approvals/{approval_id}"
    }
  ]
}
```

**Copywriting:**
- Header: `"Warning: Reminder: Approval Required"` (with Warning emoji as specified in D-17)
- Remaining time: integer minutes (e.g., `"12 minutes"`)
- If remaining time <= 5 minutes: append `" (EXPIRING SOON)"` to the "Expires In" value

## Follow-Up Message Contracts

Per D-09, the bot posts new follow-up messages (NOT in-place card edits) after approval decisions and as conversational responses.

### Approval Follow-Up Messages

| Event | Message Text | Format |
|-------|-------------|--------|
| Approved | `"Approved by {operator_upn}. Remediation is now running."` | Plain text Teams message |
| Rejected | `"Rejected by {operator_upn}. Proposal closed."` | Plain text Teams message |
| Expired | `"Approval expired for: {action_description}. No action was taken."` | Plain text Teams message |

### Conversation Response Messages

| Scenario | Message Pattern |
|----------|----------------|
| Normal response | Full triage/investigation text from Foundry as a single Teams reply (D-05) |
| 30-second interim | `"Still working on this - complex investigation in progress..."` (D-06) |
| 120-second timeout | `"The investigation is taking longer than expected. Check the Web UI for full results: {WEB_UI_PUBLIC_URL}/incidents/{incident_id}"` (D-06) |

## Internal Notify Endpoint Contract

The api-gateway calls `POST /teams/internal/notify` on the Teams bot Container App (internal VNet, not public) to trigger proactive card posting.

### Request Schema

```typescript
interface NotifyRequest {
  card_type: "alert" | "approval" | "outcome" | "reminder";
  channel_id: string;         // Teams channel ID to post to
  payload: AlertPayload | ApprovalPayload | OutcomePayload | ReminderPayload;
}

interface ReminderPayload {
  approval_id: string;
  thread_id: string;
  original_action_description: string;
  target_resources: string[];
  risk_level: "critical" | "high";
  created_at: string;         // ISO 8601 — when approval was created
  expires_at: string;         // ISO 8601 — used to compute remaining_minutes
}

interface AlertPayload {
  incident_id: string;
  alert_title: string;
  resource_name: string;
  severity: "Sev0" | "Sev1" | "Sev2" | "Sev3";
  subscription_name: string;
  domain: string;
  timestamp: string;          // ISO 8601
}

interface ApprovalPayload {
  approval_id: string;
  thread_id: string;
  proposal: {
    description: string;
    target_resources: string[];
    estimated_impact: string;
    reversibility: string;
  };
  risk_level: "critical" | "high";
  expires_at: string;         // ISO 8601
}

interface OutcomePayload {
  incident_id: string;
  approval_id: string;
  action_description: string;
  outcome_status: "Succeeded" | "Failed" | "Aborted";
  duration_seconds: number;
  resulting_resource_state: string;
  approver_upn: string;
  executed_at: string;        // ISO 8601
}
```

### Response Schema

```typescript
interface NotifyResponse {
  ok: boolean;
  message_id?: string;        // Teams message ID (for logging, not tracking)
  error?: string;
}
```

## Escalation Scheduler Contract

Per D-15 and D-16.

| Property | Value |
|----------|-------|
| Poll interval | Every 2 minutes (`setInterval`) |
| Default escalation threshold | 15 minutes (`ESCALATION_INTERVAL_MINUTES` env var) |
| Query endpoint | `GET /api/v1/approvals?status=pending` on api-gateway |
| Action | Post reminder card (Card Type 4) for each pending approval older than threshold |
| Dedup | Skip if reminder was already posted within the current escalation interval (track last reminder timestamp per approval_id in memory) |

## Bot Service Layout Contract

Per D-01, the Teams bot lives in `services/teams-bot/` as a TypeScript Container App.

```
services/teams-bot/
  src/
    index.ts              # Express server entry point
    bot.ts                # TeamsBot activity handler (extends TeamsActivityHandler)
    cards/
      alert-card.ts       # buildAlertCard(payload: AlertPayload): AdaptiveCard
      approval-card.ts    # buildApprovalCard(payload: ApprovalPayload): AdaptiveCard
      outcome-card.ts     # buildOutcomeCard(payload: OutcomePayload): AdaptiveCard
      reminder-card.ts    # buildReminderCard(payload: ReminderPayload): AdaptiveCard
    routes/
      notify.ts           # POST /teams/internal/notify handler
      health.ts           # GET /health liveness probe
    services/
      gateway-client.ts   # HTTP client for api-gateway (chat, incidents, approvals)
      escalation.ts       # Background escalation scheduler loop
      auth.ts             # Managed identity token acquisition for api-gateway calls
    config.ts             # Environment variable parsing and validation
    types.ts              # Shared TypeScript interfaces (NotifyRequest, payloads)
  Dockerfile
  package.json
  tsconfig.json
```

**File size budget:** Each card builder file should be 40-80 lines. `bot.ts` should be under 200 lines. `gateway-client.ts` under 150 lines. `escalation.ts` under 100 lines.

## Copywriting Contract

Adapted for Adaptive Card context (no empty states, no traditional CTAs — cards are event-driven, not user-navigated).

| Element | Copy |
|---------|------|
| Alert card title pattern | `"Alert: {alert_title}"` |
| Approval card title pattern | `"Remediation Approval Required ({RISK_LEVEL})"` |
| Outcome success title | `"Remediation Succeeded"` |
| Outcome failure title | `"Remediation Failed"` |
| Outcome abort title | `"Remediation Aborted"` |
| Reminder header | `"Warning: Reminder: Approval Required"` |
| Approve button | `"Approve"` |
| Reject button | `"Reject"` |
| Investigate button | `"Investigate in Web UI"` |
| View details button | `"View Details in Web UI"` |
| View in Web UI button | `"View in Web UI"` |
| 30s interim message | `"Still working on this - complex investigation in progress..."` |
| 120s timeout message | `"The investigation is taking longer than expected. Check the Web UI for full results: {url}"` |
| Approved follow-up | `"Approved by {upn}. Remediation is now running."` |
| Rejected follow-up | `"Rejected by {upn}. Proposal closed."` |
| Expired follow-up | `"Approval expired for: {description}. No action was taken."` |
| Expiring soon suffix | `"{N} minutes (EXPIRING SOON)"` (when remaining <= 5 min) |

**Destructive actions:**
- Reject is the only destructive action in card context
- Reject uses `"style": "destructive"` on the Action.Http button
- No additional confirmation modal (Teams does not support modals on Action.Http)
- The `{{userName}}` template in the Action.Http body captures the actor for audit
