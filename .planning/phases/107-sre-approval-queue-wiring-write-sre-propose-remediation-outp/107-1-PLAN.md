# Plan 107-1: Wire SRE propose_remediation to Cosmos approvals container

## Objective

`propose_remediation` in `agents/sre/tools.py` currently returns a dict but never writes to the Cosmos
`approvals` container. This means SRE-initiated remediation proposals are log-only and never appear in
the human-in-the-loop `ApprovalQueueCard` UI (which reads `GET /api/v1/approvals?status=pending`).

This plan wires `propose_remediation` to call `shared.approval_manager.create_approval_record`
(the same pattern used by `agents/compute/tools.py`) so approvals are persisted and surfaced in the UI.

It also fixes the latent `container=None` bug in `approval_manager.py` — when `container` is `None`,
the function must self-initialise a Cosmos client using `COSMOS_ENDPOINT` + `DefaultAzureCredential`
(matching the pattern in `services/api-gateway/approvals.py::_get_approvals_container`).

This means the coroutine is returned as-is (not awaited), and `record.get("id", "")` will fail at
runtime — but it is the current live pattern. To avoid breaking the contract, replicate it faithfully.
A separate cleanup phase can fix the async/sync mismatch across all agents.
</action>
