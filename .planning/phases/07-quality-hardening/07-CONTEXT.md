## Phase Boundary

Make the Azure Agentic Platform production-ready:

1. **Playwright E2E suite** — 5 new tests (incident flow, HITL Teams approval, cross-subscription RBAC, SSE reconnect, audit export) running against real deployed Container Apps in CI; existing sc1–sc6 tests refactored from mocks to real endpoints.

<decisions>
## Implementation Decisions

### E2E Test Strategy (E2E-001 through E2E-005)

- **D-01:** E2E tests run against a **real deployed environment** (not localhost, not mocks). CI spins up a dedicated E2E environment before tests and tears it down after. All 5 new E2E tests target live Container Apps endpoints. This is non-negotiable per SC-1: "no test targets localhost or stubs Azure APIs."
- **D-02:** E2E authentication uses a **dedicated service principal (client credentials flow)**. A test-scoped Entra app registration with `client_id` + `client_secret` stored in GitHub Secrets. No MSAL browser redirects in headless Playwright — the test acquires a Bearer token directly and injects it into API calls or uses `page.setExtraHTTPHeaders`.
- **D-03:** Test data isolation via **dedicated Cosmos DB containers** (`incidents-e2e`, `approvals-e2e`). A beforeAll/afterAll Playwright fixture creates these containers in the E2E environment and wipes them after each test run. Prod and staging Cosmos DB containers are never touched by E2E tests.
- **D-04:** HITL Teams approval E2E (E2E-003) verifies via **Graph API + direct webhook** — no full bot round-trip in CI. Test flow:
  1. Inject high-risk approval record into Cosmos DB (via api-gateway or direct SDK)
  2. Trigger `POST /teams/internal/notify` to simulate the bot receiving the card request
  3. Verify card was posted to Teams channel via Microsoft Graph `GET /v1.0/teams/{teamId}/channels/{channelId}/messages`
  4. Approve by calling `POST /api/v1/approvals/{id}/approve` directly (webhook simulation)
  5. Verify Cosmos DB approval record updated to `status: approved`
  6. Verify outcome card posted to Teams via Graph API

### Observability (OBSERVE-*)

- **D-05:** OpenTelemetry instrumentation depth: **auto-instrumentation only** (Phase 7). Use `opentelemetry-instrument` for Python FastAPI services and `@opentelemetry/auto-instrumentations-node` for TypeScript Express services. No manual span creation. Application Insights (`azure-monitor-opentelemetry`) as the OTLP exporter for all services.
- **D-06:** Operators view observability data in a **new "Observability" tab in the Web UI** (`services/web-ui/`). The tab queries Application Insights via the Azure Monitor Query API (`GET /subscriptions/{sub}/resourceGroups/{rg}/providers/microsoft.insights/components/{name}/api/query`). Surfaces: agent call latency (P50/P95), incident pipeline lag, approval queue depth, active errors. Read-only view — no alerts management.
- **D-07:** Trace correlation: **correlation ID injected at api-gateway ingress** via `x-correlation-id` header (already present in api-gateway code). All downstream services propagate this header. Application Insights W3C `traceparent` standard for cross-service correlation.

### Runbook Library (TRIAGE-005 support)

- **D-08:** Runbook content: **Claude generates ~10 synthetic runbooks per domain** (compute, network, storage, security, arc, sre = ~60 total). Based on CLAUDE.md domain knowledge and common Azure incident patterns. Stored as markdown files in `scripts/seed-runbooks/runbooks/` with frontmatter (`title`, `domain`, `version`, `tags`).
- **D-09:** Seed pipeline: `python scripts/seed-runbooks/seed.py` — reads markdown files, calls Azure OpenAI `text-embedding-3-small` to generate 1536-dim vectors, inserts into `runbooks` table via `psycopg` + pgvector. **Idempotent** (checks by title before inserting; re-embeds if version changed). Runs in CI on staging deploy. **Never runs against prod automatically** — prod seed is a one-time manual step documented in ops runbook.
- **D-10:** E2E-quality bar for runbooks: each runbook must return cosine similarity > 0.75 for at least one test query matching its domain (per TRIAGE-005 SC-3). The seed script validates this after insertion.

### Terraform Prod Deployment (INFRA-008)

- **D-11:** **No APIM in Phase 7.** The api-gateway Container App continues as direct public ingress. APIM Standard v2 deferred to a future milestone (no production traffic to justify ~$700/month cost; evaluated if multi-tenant or enterprise API monetization becomes a requirement).
- **D-12:** `terraform apply` on `terraform/envs/prod` applies the full module stack: Container Apps environment, Cosmos DB, PostgreSQL, Foundry workspace, VNet + private endpoints, Key Vault, ACR, App Insights. Uses **auto-generated Container Apps FQDNs** — no custom domain or Front Door in Phase 7.
- **D-13:** Prod apply is gated behind a GitHub Actions manual approval step (`environment: production` with required reviewers). CI runs `terraform plan` on PR; `terraform apply` only after human approval. Matches the Phase 1 Terraform CI/CD pattern.
- **D-14:** `services/teams-bot/` Container App registration added to Terraform prod environment. Follows the pattern established by `services/api-gateway/` and `services/web-ui/` in existing Terraform modules.

### Security Review

- **D-15:** Security review scope: **OWASP Top 10** across all HTTP-exposed services (api-gateway, web-ui, teams-bot). Automated: `bandit` for Python, `npm audit` for TypeScript. Manual review checklist covering: no hardcoded secrets, all endpoints require auth, SQL injection prevention (parameterized queries), rate limiting enabled, CORS configured.

- Web UI Observability tab component design and chart library choice
- Playwright fixture structure for E2E test data setup/teardown
- Exact OWASP checklist items beyond the core items listed above
- Runbook markdown schema/frontmatter format

<specifics>
## Specific Ideas

- **HITL E2E via Graph API**: Verify Teams card posting by checking the channel message history via `GET /v1.0/teams/{id}/channels/{id}/messages` — this gives a real assertion without needing a bot client in CI.
- **E2E environment teardown**: Use a Playwright global teardown (`globalSetup`/`globalTeardown`) to clean the `incidents-e2e` and `approvals-e2e` Cosmos DB containers. Keep the teardown idempotent (safe to run twice).
- **Runbook quality bar**: Each of the ~60 runbooks should include: symptom description, root cause, diagnostic steps (3-5), remediation commands (agent-executable), and rollback procedure. Enough content for high cosine similarity on natural-language queries.

</specifics>

<deferred>
## Deferred Ideas

- **APIM Standard v2** — Deferred to future milestone. Revisit when multi-tenant or API monetization becomes a requirement. Phase 6 D-03 explicitly noted "evaluate at Phase 7 planning with production traffic data" — no production traffic yet, decision is to skip.
- **Custom domain + TLS** — No branded URL for prod in Phase 7. Use auto-generated Container Apps FQDNs. Add custom domain via Azure Front Door in a future hardening pass.
- **Full bot round-trip E2E** — Teams approval E2E uses Graph API verification instead of a full bot client in CI. Full bot round-trip (sending messages via Teams Bot Connector API) deferred to Phase 8 if needed.
- **Manual OpenTelemetry spans** — Auto-instrumentation only in Phase 7. Manual agent-level spans (per-Foundry-call, per-tool-call latency) are a Phase 8 observability improvement.

</deferred>
