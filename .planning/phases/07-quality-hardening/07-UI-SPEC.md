# Phase 7 — UI Design Contract

> Visual and interaction contract for the Quality & Hardening Observability tab. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** One new UI surface — an **Observability tab** added to the existing `DashboardPanel` right-pane TabList in `services/web-ui/`. The tab displays agent call latency (P50/P95), incident pipeline lag, approval queue depth, and active errors by querying the Azure Monitor Query API. It is **read-only** — no write actions, no destructive operations.
>
> **Design continuity:** This phase inherits 100% of the Phase 5 design system (Fluent UI v9, Griffel, spacing tokens, typography presets, color tokens). No new design tokens are introduced. All new components follow the established patterns in `05-UI-SPEC.md`.

## Component Inventory

All Phase 7 components are built on Fluent v9 primitives. No new third-party libraries.

### Observability Tab Components

| Component | Fluent v9 Base | Props | Behavior |
|-----------|---------------|-------|----------|
| `ObservabilityTab` | `div` + `makeStyles` | `subscriptions: string[]` | Container component. Fetches all observability data via a single Next.js API route (`GET /api/observability?timeRange={value}`). Manages polling (30s interval). Renders 4 metric card children. Shows `lastUpdated` timestamp in `Caption` and time range `Dropdown`. |
| `MetricCard` | `Card` variant="filled-alternative" + `Text` | `title: string`, `children: ReactNode`, `health: 'healthy' \| 'warning' \| 'critical'` | Reusable card wrapper. Background: `tokens.colorNeutralBackground3`. Border-left: 3px solid using health color token (green/yellow/red). Title in `Subtitle` (16px semibold). Health badge in top-right corner: `Badge` with `appearance="filled"` and semantic color. |
| `AgentLatencyCard` | `MetricCard` + `DataGrid` | `data: AgentLatencyRow[]` | 7-row grid (one per agent). Columns: Agent Name (`Body`), P50 (`Caption` monospace), P95 (`Caption` monospace). P95 cell text color uses semantic health color based on threshold. Overall card health = worst P95 value across agents. |
| `PipelineLagCard` | `MetricCard` + `Text` list | `data: PipelineLagData` | Three labeled rows: "Alert to Incident", "Incident to Triage", "Total E2E". Each row: label (`Body`) + value (`Caption` monospace, right-aligned). Overall card health based on Total E2E lag vs thresholds. |
| `ApprovalQueueCard` | `MetricCard` + `Text` list | `data: ApprovalQueueData` | Two labeled rows: "Pending" (count), "Oldest" (relative time). Card health based on pending count vs thresholds. |
| `ActiveErrorsCard` | `MetricCard` + expandable list | `data: ActiveError[]` | Expandable error list using Fluent `Accordion` + `AccordionItem`. Each item: timestamp (`Caption`) + error summary (`Body`). Expanding shows error detail in `tokens.fontFamilyMonospace` 12px (matching TraceTree pattern). Card health: critical if > 0 errors, healthy if 0. |
| `TimeRangeSelector` | `Dropdown` | `value: string`, `onChange: (value: string) => void` | Single-select dropdown. Options: "Last 1 hour", "Last 6 hours", "Last 24 hours", "Last 7 days". Default: "Last 1 hour". Width: 160px. |

### Data Flow

```
ObservabilityTab
  |-- GET /api/observability?timeRange=1h (Next.js API route)
        |-- Azure Monitor Query API (server-side, DefaultAzureCredential)
              |-- Application Insights: agent latency KQL
              |-- Application Insights: pipeline lag KQL
              |-- Application Insights: active errors KQL
        |-- Cosmos DB query (server-side)
              |-- Pending approvals count + oldest timestamp
```

The Next.js API route (`app/api/observability/route.ts`) runs server-side to:
1. Authenticate to Azure Monitor Query API via `DefaultAzureCredential`
2. Execute KQL queries against Application Insights for latency, lag, and errors
3. Query Cosmos DB for approval queue depth
4. Return a single JSON response combining all metrics

## Interaction Patterns

### Observability Tab

| State | Visual | Trigger |
|-------|--------|---------|
| Loading | 4x `Skeleton` cards (matching MetricCard dimensions), each with 3 `SkeletonItem` rows | Tab first selected, or time range changed |
| Populated | 4 MetricCards with live data, health badges, `lastUpdated` timestamp | Successful API response |
| Auto-refresh | No visual indicator during refresh; data updates in-place without flash | 30-second polling interval |
| Empty (no data) | MetricCard shows "No data for this time range" in `Body` text, centered | Query returns empty results |
| Error (API failure) | Fluent `MessageBar` intent="error" above cards: error state copy (see Copywriting) | API route returns non-200 or network failure |
| Partial failure | Cards with data render normally; failed cards show individual "Unable to load" state | One query succeeds, another fails |

### Health Badge Transitions

Health badges update on each 30-second poll. Color transitions are instant (no animation) — the `Badge` component re-renders with the new `color` prop. No flashing or attention-grabbing animation for health changes; the operator notices via color change on their next glance.

## Registry Safety

| Registry | Blocks Used | Safety Gate |
|----------|-------------|-------------|
| Fluent UI v9 (official) | `Card`, `DataGrid`, `Badge`, `Accordion`, `AccordionItem`, `Dropdown`, `Skeleton`, `SkeletonItem`, `MessageBar`, `Text`, `Tab` (all from `@fluentui/react-components`) | not required (Microsoft-maintained, GA) |
| `@fluentui/react-icons` | `PulseRegular` | not required (Microsoft-maintained, GA) |

**No new third-party packages.** No chart library added — Phase 7 observability uses Fluent v9 `DataGrid` for tabular latency data and `Text` for scalar metrics. This avoids introducing a charting dependency (e.g., recharts, visx) for 4 simple metric displays. If richer visualizations are needed in a future phase, a charting library can be evaluated then.

**Rationale for no chart library (Claude's Discretion per CONTEXT.md):**
- Agent latency is best displayed as a 7-row table (one row per agent, P50 + P95 columns) — a bar chart adds visual complexity without improving readability for 7 data points
- Pipeline lag is 3 scalar values — text display is clearer than a chart
- Approval queue depth is 2 scalar values — text display is optimal
- Active errors is a list — expandable accordion is the right pattern
- Adding recharts (~450KB gzipped) or visx (~200KB gzipped) for these use cases would be over-engineering

**Source:** CONTEXT.md "Claude's Discretion" (chart library choice)

## Accessibility

Inherited from Phase 5. Phase 7 additions:

| Requirement | Implementation |
|-------------|---------------|
| Tab keyboard navigation | Observability tab reachable via Arrow keys in TabList (Fluent v9 handles this natively). |
| Screen reader | MetricCard titles use `role="region"` with `aria-label="{card title}"`. Health badges include `aria-label="Health status: {healthy\|degraded\|critical}"`. Last updated timestamp uses `aria-live="polite"` to announce refreshes. |
| Focus management | On tab switch to Observability, focus moves to the TimeRangeSelector (first focusable element). Error accordion items are focusable via Tab key. |
| Color contrast | Health colors use `Foreground1` tokens (not `Background` tokens) for text on white backgrounds, guaranteeing WCAG 2.1 AA 4.5:1 contrast. |

## Dark Mode

Fully supported via Fluent v9 token inheritance. MetricCard surfaces use `tokens.colorNeutralBackground3` which automatically maps to the correct dark-mode value. Health badge colors use semantic tokens that adapt to dark theme. No additional CSS or token overrides needed.
