## Tasks

### Task 1: Fix `deploy-all-images.yml` (Bug 1)

**File:** `.github/workflows/deploy-all-images.yml`

Two sub-fixes:

**1a. Add missing `AZURE_CLIENT_SECRET` to `build-azure-mcp-server` job (line ~301-304)**

The `build-azure-mcp-server` job only passes 3 secrets (`AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, `AZURE_SUBSCRIPTION_ID`). All other service build jobs pass 4 including `AZURE_CLIENT_SECRET`. The reusable `docker-push.yml` uses `AZURE_CLIENT_SECRET` for its auth mode detection â€” without it, the job always falls through to OIDC even when client-secret auth is configured.

Fix: Add `AZURE_CLIENT_SECRET: ${{ secrets.AZURE_CLIENT_SECRET }}` to the `secrets:` block.

**1b. Use resolved image tag for all service builds (lines ~244, 258, 271, 286, 300)**

Service builds (`build-api-gateway`, `build-arc-mcp-server`, `build-teams-bot`, `build-web-ui`, `build-azure-mcp-server`) pass `image_tag: ${{ inputs.image_tag }}` directly. When `inputs.image_tag` is empty (the default), this passes an empty string to docker-push.yml â€” which then resolves to `github.sha` internally. This works but is fragile and inconsistent with agent builds which use the resolved tag from `build-agent-base.outputs.image_tag`.

Fix: Add `needs: build-agent-base` to all 5 service build jobs and change `image_tag` to `${{ needs.build-agent-base.outputs.image_tag }}`. This ensures all images (agents + services) are tagged identically from a single resolution point. Also add them to the dependency chain so the resolved tag is available.

- [ ] Add `AZURE_CLIENT_SECRET` to `build-azure-mcp-server` secrets block
- [ ] Add `needs: build-agent-base` to all 5 service build jobs
- [ ] Change `image_tag` to `${{ needs.build-agent-base.outputs.image_tag }}` for all 5 service builds

### Task 2: Fix `terraform-apply.yml` (Bugs 2 & 3)

**File:** `.github/workflows/terraform-apply.yml`

Three psql steps (dev line ~73, staging line ~148, prod line ~248) are missing:
- `PGSSLMODE: require` in the `env:` block â€” Azure PostgreSQL Flexible Server requires SSL; without this the connection may fail or use an insecure channel. The `prod-db-setup.yml` workflow correctly sets this (line 129).
- `-v ON_ERROR_STOP=1` flag â€” without this, SQL errors (e.g., extension creation failure) are silently swallowed and the step reports success.

Fix: Add `PGSSLMODE: require` to the `env:` block and add `-v ON_ERROR_STOP=1` to the `psql` command for all 3 "Create pgvector Extension" steps.

- [ ] Dev: Add `PGSSLMODE: require` and `-v ON_ERROR_STOP=1`
- [ ] Staging: Add `PGSSLMODE: require` and `-v ON_ERROR_STOP=1`
- [ ] Prod: Add `PGSSLMODE: require` and `-v ON_ERROR_STOP=1`

### Task 3: Fix `staging-e2e-simulation.yml` (Bug 4) + Assess Bug 5

**File:** `.github/workflows/staging-e2e-simulation.yml`

**Bug 4:** The e2e job (lines 52-54) runs `npm init -y` which overwrites the committed `e2e/package.json`, then installs specific packages at non-deterministic latest versions. The `e2e/` directory already has a committed `package.json` and `package-lock.json`.

Fix:
- Replace `npm init -y` + `npm install ...` with `npm ci` (uses lockfile for deterministic installs)
- Add `cache: npm` and `cache-dependency-path: e2e/package-lock.json` to the `actions/setup-node@v4` step for faster CI runs

Note: `e2e/package.json` has `@azure/monitor-query` missing from devDependencies but the old workflow installed it. Check if any e2e test actually imports it; if so, add it to `e2e/package.json` as part of this fix before `npm ci` can work.

**Bug 5 (agent-images.yml) â€” NOT A BUG (will document, not fix):**

After careful analysis, the deploy jobs' `image_tag: ${{ github.sha }}` in `agent-images.yml` is actually **correct**. The build jobs do NOT pass `image_tag` to docker-push.yml, so docker-push.yml's "Resolve image tag" step defaults to `github.sha` (lines 122-125 of docker-push.yml). The `resolve-base-image.outputs.tag` is only used for the `BASE_IMAGE` build arg (the FROM layer), not for the output image tag. Changing deploy jobs to use `resolve-base-image.outputs.tag` would deploy an image tag that doesn't exist (it's a stale base image SHA, not the current build SHA).

- [ ] Add `cache: npm` + `cache-dependency-path` to setup-node step
- [ ] Replace `npm init -y` + `npm install` with `npm ci`
- [ ] Check if `@azure/monitor-query` is imported by e2e tests; if so, add to `e2e/package.json`
- [ ] Document Bug 5 as non-bug in this plan (DONE above)
