# Quick Plan: Add Structured Logging to Web UI Next.js API Routes

**ID:** 260401-o1l
**Type:** quick
**Created:** 2026-04-01
**Branch:** `quick/260401-o1l-web-ui-structured-logging`

---

## Problem

All 13 Next.js API route handlers in `services/web-ui/app/api/` have **zero logging**. When these routes run as a Container App, errors are completely invisible in Azure Log Analytics. The SSE stream route (`/api/stream`) is the worst blind spot — it has complex Foundry polling logic with silent `catch {}` blocks that swallow network errors, status transitions, and timeouts.

The Python agent containers already have structured logging via `agents/shared/logging_config.py`. The web-ui needs an equivalent TypeScript pattern.

## Approach

Create a lightweight structured JSON logger (`lib/logger.ts`) and add it to all 13 API routes. No new npm dependencies — just `console.log` (stdout) and `console.error` (stderr) with JSON formatting, which Azure Container Apps captures into `ContainerAppConsoleLogs_CL`.

---

## Tasks

### Task 1: Create `lib/logger.ts` — structured JSON logger

**New file:** `services/web-ui/lib/logger.ts`

- JSON-line format: `{ timestamp, level, msg, service: "aap-web-ui", ...ctx }`
- 4 levels: `debug`, `info`, `warn`, `error`
- `LOG_LEVEL` env var support (default: `"info"`); debug suppressed in prod
- `error` level writes to `console.error` (stderr); all others to `console.log` (stdout)
- Server-side only header comment: `// Server-side only — do not import in client components`
- Helper: `logger.child({ route: "/api/stream" })` returns a scoped logger that auto-includes `route` in every entry (reduces boilerplate in routes)

**Acceptance:**
- [ ] `lib/logger.ts` exports `logger` object with `debug`, `info`, `warn`, `error` methods
- [ ] Each method accepts `(msg: string, ctx?: Record<string, unknown>)`
- [ ] Output is valid JSON per line
- [ ] LOG_LEVEL filtering works (debug < info < warn < error)
- [ ] `child()` method returns logger with merged context
- [ ] No npm dependencies added

### Task 2: Instrument all 13 API route handlers

**Files to modify (13 routes):**

| Route | Key logging points |
|---|---|
| `app/api/stream/route.ts` | SSE connect (thread_id, run_id), each poll status transition, not_found count, timeout, network errors in catch, connection close/abort |
| `app/api/proxy/chat/route.ts` | Request start (has thread_id?), upstream error (status + detail), network error, success |
| `app/api/proxy/chat/result/route.ts` | Poll request (thread_id), upstream error, success (run_status) |
| `app/api/proxy/incidents/route.ts` | Request start (query params), upstream error, success (count) |
| `app/api/proxy/patch/assessment/route.ts` | Request start, upstream error, success |
| `app/api/proxy/patch/installations/route.ts` | Request start, upstream error, success |
| `app/api/proxy/patch/installed/route.ts` | Request start, upstream error, success |
| `app/api/proxy/approvals/[approvalId]/approve/route.ts` | Approval action (approvalId), upstream error, success |
| `app/api/proxy/approvals/[approvalId]/reject/route.ts` | Reject action (approvalId), upstream error, success |
| `app/api/observability/route.ts` | Query start (timeRange), individual KQL/Cosmos errors, success |
| `app/api/resources/route.ts` | Request start (subscriptions, type filter), ARM errors, success (count) |
| `app/api/subscriptions/route.ts` | Request start, ARM error, success (count) |
| `app/api/topology/route.ts` | Request start, ARM error, success |

**Logging pattern per proxy route (consistent):**
```typescript
import { logger } from '@/lib/logger';
const log = logger.child({ route: '/api/proxy/chat' });

// Request start
log.info('proxy request', { method: 'POST', thread_id });

// Upstream error
log.error('upstream error', { status: res.status, detail: data?.detail });

// Network/timeout error
log.error('gateway unreachable', { error: message });

// Success (debug to avoid noise)
log.debug('proxy response', { status: res.status });
```

**SSE stream route gets extra logging:**
```typescript
log.info('sse connect', { thread_id, run_id, stream_type });
log.debug('poll status', { run_status, not_found_count });
log.warn('poll timeout', { thread_id, elapsed_ms });
log.error('poll error', { thread_id, error: message });
log.info('sse close', { thread_id, final_status, seq });
```

**Acceptance:**
- [ ] All 13 routes import logger and log request start + errors
- [ ] SSE stream route logs connect, poll transitions, timeout, close
- [ ] No logging of sensitive data (no auth headers, no tokens, no full request bodies)
- [ ] `npx tsc --noEmit` passes
- [ ] No client components import logger

- NO new npm dependencies
- Server-side only — never imported in `components/` or client-side files
- JSON output per line (machine-readable for Log Analytics queries)
- LOG_LEVEL env var (default: "info") — aligns with Python agent pattern
- No logging of secrets, auth headers, or PII
