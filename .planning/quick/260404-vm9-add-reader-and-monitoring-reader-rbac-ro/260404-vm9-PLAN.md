# Quick Plan: Add Reader and Monitoring Reader RBAC for API Gateway

## Problem

The API gateway managed identity (`ca-api-gateway-prod`, principal `69e05934-...`) is missing two RBAC roles needed for VM health and metrics endpoints:

1. **Reader** at subscription scope - needed for `Microsoft.ResourceHealth/availabilityStatuses/read`
2. **Monitoring Reader** at subscription scope - needed for `Microsoft.Insights/metrics/read`

The gateway's diagnostic pipeline (`diagnostic_pipeline.py`) calls `_collect_resource_health()` and `_collect_metrics()` which require these roles. Currently the gateway only has `Cognitive Services User` + `Azure AI Developer` + `Cosmos DB Operator`.

Other agents (compute, sre, patch) already have these roles via the same RBAC module - the API gateway was simply never given them.

## Analysis

**File:** `terraform/modules/rbac/main.tf`

**Existing pattern:** The file uses a `local.role_assignments` map with `merge()` blocks keyed by `"{agent}-{role}-{scope}"`. The `for_each` on `azurerm_role_assignment.agent_rbac` iterates over this flat map.

**Scope decision:** The API gateway needs Reader + Monitoring Reader on **all subscriptions** (via `var.all_subscription_ids`), not just the platform subscription. Reason: VM health and metrics queries target VMs that may live in any monitored subscription (compute, network, storage subscriptions). This matches the pattern used by SRE and Patch agents which also do cross-subscription reads.

**No new variables needed.** `var.all_subscription_ids` and `var.agent_principal_ids["api-gateway"]` are already available.

## Tasks

- [ ] **Task 1: Add Reader + Monitoring Reader role assignments for api-gateway**
  - **File:** `terraform/modules/rbac/main.tf`
  - **Action:** Add a new `merge()` block (after the existing api-gateway blocks, before the "All agents: Cosmos DB Operator" block) following the exact same pattern as SRE/Patch agents:
    ```hcl
    # API Gateway: Reader + Monitoring Reader across all in-scope subscriptions
    # (VM health checks via ResourceHealth API + Azure Monitor metrics reads)
    merge(
      {
        for sub_id in var.all_subscription_ids :
        "api-gateway-reader-${replace(sub_id, "-", "")}" => {
          principal_id         = var.agent_principal_ids["api-gateway"]
          role_definition_name = "Reader"
          scope                = "/subscriptions/${sub_id}"
        }
      },
      {
        for sub_id in var.all_subscription_ids :
        "api-gateway-monreader-${replace(sub_id, "-", "")}" => {
          principal_id         = var.agent_principal_ids["api-gateway"]
          role_definition_name = "Monitoring Reader"
          scope                = "/subscriptions/${sub_id}"
        }
      }
    ),
    ```
  - **Placement:** Insert after line 138 (closing `}` of the api-gateway Azure AI Developer block), before the "All agents: Cosmos DB Operator" comment on line 140.

- [ ] **Task 2: Validate**
  - Run `terraform fmt -check` on the module
  - Run `terraform validate` (or at minimum `terraform fmt`) to ensure syntax is correct
  - Verify no duplicate keys would be produced (api-gateway-reader-* and api-gateway-monreader-* are unique prefixes not used elsewhere)

## Risk Assessment

- **Low risk.** This is additive-only (new role assignments). No existing resources modified.
- **No state conflicts.** New map keys (`api-gateway-reader-*`, `api-gateway-monreader-*`) are unique.
- **Follows proven pattern.** Identical structure to SRE and Patch agent blocks.
- **Operator step required:** `terraform apply` after merge to create the role assignments in Azure.
