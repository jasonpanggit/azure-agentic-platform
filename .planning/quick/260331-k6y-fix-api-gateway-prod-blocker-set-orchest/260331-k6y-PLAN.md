| # | Blocker | Root cause |
|---|---------|-----------|
| B1 | `ORCHESTRATOR_AGENT_ID` missing on `ca-api-gateway-prod` | `foundry.py` raises `ValueError` on every chat request |
| B2 (F-01) | `Azure AI Developer` RBAC not assigned to gateway MI | Gateway gets 403 from Foundry agent/thread/run APIs |

**Terraform note (CRITICAL):** `main.tf` comments warn that any env vars set manually via `az containerapp update` will be wiped on the next `terraform apply` unless also set in `terraform.tfvars`. Must update tfvars and the RBAC module so fixes survive a re-apply.

## Tasks

### Task 1 — Set ORCHESTRATOR_AGENT_ID env var on ca-api-gateway-prod

**Why:** `foundry.py:create_foundry_thread()` reads `os.environ.get("ORCHESTRATOR_AGENT_ID")` and raises `ValueError` if missing, causing every `/api/v1/chat` SSE stream to fail immediately.

**Steps:**

1. Set the env var via `az containerapp update`:
   ```bash
   az containerapp update \
     --name ca-api-gateway-prod \
     --resource-group rg-aap-prod \
     --set-env-vars "ORCHESTRATOR_AGENT_ID=asst_NeBVjCA5isNrIERoGYzRpBTu"
   ```

2. Wire it in `terraform/envs/prod/terraform.tfvars` so it survives the next `terraform apply`:
   ```hcl
   orchestrator_agent_id = "asst_NeBVjCA5isNrIERoGYzRpBTu"
   ```
   > Variable `orchestrator_agent_id` is already declared in `terraform/envs/prod/variables.tf:114` and passed to the agent-apps module at `main.tf:198`. Adding it to tfvars is all that's needed.

3. Verify the env var is live:
   ```bash
   az containerapp show \
     --name ca-api-gateway-prod \
     --resource-group rg-aap-prod \
     --query "properties.template.containers[0].env[?name=='ORCHESTRATOR_AGENT_ID'].value" \
     --output tsv
   ```
   Expected output: `asst_NeBVjCA5isNrIERoGYzRpBTu`

### Task 2 — Grant Azure AI Developer RBAC to gateway managed identity

**Why:** The RBAC module only grants `Cognitive Services User` to the gateway MI (for inference). `Azure AI Developer` is required for agent management APIs (`/agents`, `/threads`, `/runs`). Without it, Foundry returns 403 on every thread/run creation.

**Steps:**

1. Create the role assignment immediately via `az`:
   ```bash
   az role assignment create \
     --assignee "69e05934-1feb-44d4-8fd2-30373f83ccec" \
     --role "Azure AI Developer" \
     --scope "/subscriptions/4c727b88-12f4-4c91-9c2b-372aab3bbae9/resourceGroups/rg-aap-prod/providers/Microsoft.CognitiveServices/accounts/aap-foundry-prod"
   ```

2. Add the assignment to `terraform/modules/rbac/main.tf` so it's managed as IaC and won't regress. In the `# API Gateway` block (around line 121), add alongside the existing `Cognitive Services User` entry:
   ```hcl
   # API Gateway: Azure AI Developer on Foundry account (agent/thread/run APIs)
   "api-gateway-aidev-foundry" = {
     principal_id         = var.agent_principal_ids["api-gateway"]
     role_definition_name = "Azure AI Developer"
     scope                = "/subscriptions/${var.platform_subscription_id}/resourceGroups/${var.resource_group_name}/providers/Microsoft.CognitiveServices/accounts/${var.foundry_account_name}"
   }
   ```
   > This requires `resource_group_name` and `foundry_account_name` variables — check `terraform/modules/rbac/variables.tf` to see if they already exist; add them if not.

3. Verify the role assignment exists:
   ```bash
   az role assignment list \
     --assignee "69e05934-1feb-44d4-8fd2-30373f83ccec" \
     --scope "/subscriptions/4c727b88-12f4-4c91-9c2b-372aab3bbae9/resourceGroups/rg-aap-prod/providers/Microsoft.CognitiveServices/accounts/aap-foundry-prod" \
     --query "[].roleDefinitionName" \
     --output tsv
   ```
   Expected output includes: `Azure AI Developer`
